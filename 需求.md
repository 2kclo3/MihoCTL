**项目名称：MihoCTL**

**【项目背景与目标】** 你现在是一位资深的系统级开发工程师。我们需要从零开始开发一个名为 `MihoCTL` 的命令行工具（CLI）。 该工具主要用于在无头服务器（Headless Server，主要环境为 Linux 和 macOS）上管理、控制和配置 Mihomo（Clash.Meta 的分支）代理内核。 我们的目标是遵循 Unix 哲学，打造一个轻量、高效、无外部环境依赖且支持交叉编译的单文件 CLI 工具。

**【技术栈要求】**

- **开发语言：** Go (Golang)，利用其优秀的跨平台编译特性。
- **CLI 框架：** `[github.com/spf13/cobra](https://github.com/spf13/cobra)`，用于构建层次化的命令结构和自动生成帮助文档。
- **HTTP 请求：** 使用 Go 标准库 `net/http` 或 `go-resty/resty`，用于与 Mihomo 的本地 RESTful API 进行交互。
- **配置管理：** `[github.com/spf13/viper](https://github.com/spf13/viper)`（可选），用于读取和管理 MihoCTL 自身的配置。
- **多语言支持：** 通过内置字典（建议 JSON/YAML 或 Go map）实现 CLI 输出文案国际化（i18n），首期支持中文（`zh-CN`）与英文（`en-US`）。

**【核心架构与职责设计】** MihoCTL 主要负责以下核心任务，请在架构设计时解耦各部分逻辑：

1. **进程与文件管理（System/File Level）：**
   - 负责拉取和更新 Mihomo 订阅的 YAML 配置文件。
   - 在 Linux 下通过 `systemd` 或在 macOS 下通过 `launchd`（或通过简单的 `exec.Command` 后台运行）来管理 Mihomo 内核的启动、停止和重启。
2. **API 状态控制（Network/API Level）：**
   - 不修改底层 YAML，而是直接通过 HTTP 调用 Mihomo 本地开放的 Controller API（默认 `127.0.0.1:9090`）。
   - 实现获取节点状态、测速、切换策略组节点等功能。
   - 支持通过 Controller API 动态切换 TUN 模式开关（尽量不直接改写订阅 YAML）。
   - 支持系统代理开关控制与状态查询（优先走 Mihomo/系统能力，失败时提供清晰降级提示）。
3. **语言与交互层（UX/I18N Level）：**
   - CLI 的帮助信息、错误提示、执行结果支持中英文切换。
   - 支持通过配置文件、环境变量或命令参数指定语言（优先级建议：命令参数 > 环境变量 > 配置文件 > 默认值）。
4. **系统开机启动管理（Boot Service Level）：**
   - 支持将 Mihomo 注册为系统级开机自启服务，并在用户登录前启动。
   - Linux 使用 `systemd` 的 system 级 unit（如 `/etc/systemd/system/mihomo.service`）。
   - macOS 使用 `launchd` 的 `LaunchDaemon`（如 `/Library/LaunchDaemons/`），而不是仅用户登录后生效的 `LaunchAgent`。

**【命令结构规划】** 请按照以下结构初始化 Cobra 命令树：

遵循现代 CLI 的分层设计原则 (资源+动作)，命令树具体规划如下：

1. 核心生命周期控制 (Top-Level)
最常用的高频操作，直接挂载在根命令下。
- `mihoctl start`：启动 Mihomo 守护进程。
- `mihoctl stop`：停止 Mihomo 守护进程。
- `mihoctl restart`：重启 Mihomo 服务。
- `mihoctl status`：输出当前服务运行状态（PID、运行时长）及 Controller API 连通性。

2. 后台服务管理 (Service)
负责操作系统级别的自启与守护注册（Linux: systemd / macOS: launchd）。
- `mihoctl service enable`：注册并启用系统开机自启（需提权）。
- `mihoctl service disable`：注销并关闭系统开机自启。
- `mihoctl service status`：查看开机自启配置的当前状态。

3. 订阅与配置管理 (Sub)
- `mihoctl sub update [url]`：下载或更新订阅配置文件（如果不带 url，则更新默认订阅）。
- `mihoctl sub add <url>`：添加新的订阅链接。
- `mihoctl sub list`：列出当前已保存的订阅源。

4. 代理与分流控制 (Proxy)
负责与 Controller API 交互，动态调整代理策略。
- `mihoctl proxy ls`：获取并格式化打印所有代理策略组及节点列表。
- `mihoctl proxy use <group> <proxy>`：切换指定策略组的节点（替代 set，语义更准确）。
- `mihoctl proxy check`：一键测试当前所有策略组节点的延迟。

5. 流量接管模式 (Mode)
管理系统的网络流量劫持方式。
- `mihoctl mode sys on`：开启系统级全局代理（环境变量/系统代理设置）。
- `mihoctl mode sys off`：关闭系统代理。
- `mihoctl mode tun on`：开启网卡级 TUN 虚拟局域网模式。
- `mihoctl mode tun off`：关闭 TUN 模式。
- `mihoctl mode status`：一览当前系统的流量接管状态（System Proxy / TUN）。

6. 全局偏好设置 (Config)
CLI 工具自身的行为配置。
- `mihoctl config set-lang <zh-CN|en-US>`：设置 CLI 终端输出语言。
- `mihoctl config view`：查看当前所有配置项（包括语言、API 端口、配置路径等）。


**【跨平台兼容性注意】** 在涉及文件路径和系统服务时，请使用 `runtime.GOOS` 进行分支处理：

- **Linux 默认路径：** 配置及内核放置在 `~/.config/mihomo/` 或 `/etc/mihomo/`。
- **macOS 默认路径：** 配置及内核放置在 `~/Library/Application Support/mihomo/`。
- **开机自启差异：**
   - Linux 使用 `systemctl enable --now mihomo` 管理 system 级服务。
   - macOS 使用 `launchctl bootstrap system /Library/LaunchDaemons/<plist>` 管理系统级守护进程。
   - 以上操作均涉及系统目录写入与服务注册，需 root 或 sudo 权限。
- **系统代理差异：**
   - Linux 发行版差异较大，需优先适配无桌面环境可用方案；若系统不支持统一代理接口，应返回明确提示并给出手动处理建议。
   - macOS 可通过系统网络服务进行代理开关控制（通常需要管理员权限或具备相应授权）。
- **TUN 模式差异：**
   - TUN 开启通常需要更高权限（如 root/sudo）与系统网络能力支持。
   - 若运行环境缺失 TUN 依赖（内核模块、权限、路由能力），命令应失败并给出可操作的错误提示。

**【语言支持要求】**

- 默认语言建议为中文（`zh-CN`），并可切换为英文（`en-US`）。
- 至少覆盖以下文案：命令帮助、参数校验错误、网络请求失败、服务状态输出、订阅更新结果。
- 所有用户可见输出应统一通过文案层获取，避免在业务逻辑中硬编码中文/英文字符串。

**【系统代理与 TUN 开关要求】**

- 需要提供显式开关命令，不依赖手工编辑配置文件。
- `system-proxy` 与 `tun` 均需支持 `on/off/status` 三类操作。
- `status` 输出需包含：当前状态、变更来源（API/系统命令）、最近一次错误（如有）。
- 对权限不足、接口不可用、平台不支持等情况，返回结构化错误信息与下一步建议。
